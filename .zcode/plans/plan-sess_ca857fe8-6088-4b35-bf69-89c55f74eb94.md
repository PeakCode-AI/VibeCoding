# 生产部署文档与脚本更新方案

## 目标
以 `docs/` 现状为统一基准,修复所有陈旧/错误的端口与路径,补齐缺失的部署资产,使各端生产部署文档与脚本一致可用。**不新增 CI/CD**。

## 统一端口基准(docs 现状)
- VibeAdmin 前端 80/5173(开发)、后端 **8080**
- VibeBase 前端 80/5175(开发)、后端 **8081**(仅此一个端口,移除文档中错误的 8881/8880)
- Vibe-Mp-H5 开发 5174、生产静态产物部署到 Nginx
- PostgreSQL 5432(项目级 compose)/ 5433(monorepo 中间件)、Redis 6379、MinIO 9000
- Compose 内部服务名:VibeAdmin 的 api 服务名为 `api`(不是 `backend`)

## A. 修复脚本与配置错误(代码级)

### A1. `VibeBase/vibe-base/Dockerfile`(端口 7860 → 8081)
- `EXPOSE 7860` → `EXPOSE 8081`
- `HEALTHCHECK` 中 `localhost:7860` → `localhost:8081`
- `CMD` 中 `--port 7860` → `--port 8081`

### A2. `VibeBase/vibe-base/docker-compose.yml`(端口 8880 清理 + 上下文修正)
- `backend` 服务 `ports`:仅保留 `"8081:8081"`,移除 `"8880:8880"`
- (上下文 `../..` + dockerfile 路径当前正确,保留)

### A3. `VibeBase/vibe-base-web/nginx.conf`(新增,修复构建失败)
参照 `VibeAdmin/vibe-admin-web/nginx.conf` 风格新增:SPA fallback + `/api/` 反代到后端服务 `backend:8081`(与 docker-compose 服务名 `backend` 对应)。

### A4. `VibeAdmin/vibe-admin-web/nginx.conf`(8000 → 8080 + 服务名)
- `proxy_pass http://backend:8000` → `proxy_pass http://api:8080`
  (compose 中 api 服务名是 `api`,且实际端口 8080)

### A5. `VibeAdmin/vibe-admin/Dockerfile`(8000 → 8080)
- `EXPOSE 8000` → `EXPOSE 8080`
- `CMD` `--port 8000` → `--port 8080`

### A6. `VibeBase/vibe-base/start_server.sh`(硬编码路径修复)
- `PROJECT_DIR` 改为相对脚本位置自动推导:`SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"` 并 `cd "$SCRIPT_DIR"`
- 移除 `--reload`(生产脚本应无 reload),保留 poetry/uvicorn 双路径
- 文件头补充说明

### A7. `VibeBase/start_dev.sh`(启 MySQL → 启 PG + 路径修复)
- 删除启动 MySQL 的逻辑(架构已迁 PG)
- 改为:依赖 `docker-compose.middleware.yml` 或提示先启 PG/Redis
- 路径改为基于脚本位置相对推导,移除硬编码 `/Users/jwangkun/Coding/VibeBase/...`
- 启动后端 8081、前端 5175
