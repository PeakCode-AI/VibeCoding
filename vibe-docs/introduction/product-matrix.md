# 产品矩阵

> 五个入口，一套生意。统一的用户、统一的账本、统一的经营、**统一的收钱**。

VibeBase 是 Vibe 产品体系的**用户端产品**。它和另外四个端共同构成了一套完整的 AI 商业化平台——从对话、运营到**支付收钱**全链路打通。理解五端的关系，是理解整个产品体系的关键。

## 五端总览

| 产品 | 定位 | 技术栈 | 面向 |
| --- | --- | --- | --- |
| **VibeBase** | C 端用户产品 | Vue 3 + FastAPI | 最终用户 |
| **VibeAdmin** | 运营后台 | Vue 3 + FastAPI | 运营 / 管理员 |
| **VibeApp** | 移动 App | Flutter | 移动端用户 |
| **Vibe-Mp-H5** | 小程序 / H5 | uni-app | 微信生态用户 |
| **VibePay** | 支付中台 | Spring Boot + 安卓监控端 | 运营 / 商户 |

## 四个统一

五端能协同工作的核心，是共享四个「统一」：

### 1. 统一的用户

四端的用户登录、注册、身份认证都基于**同一套 JWT 体系**。

- **共享密钥**：VibeBase 与 VibeAdmin 配置相同的 `SECRET_KEY`，签发的 JWT 可跨服务互认
- **共享用户表**：所有端的用户数据都落在同一张 `users` 表
- **统一角色**：`roles` / `user_roles` 表定义的角色在所有端通用

这意味着：一个用户在 Web 端注册，App 端、小程序端可直接登录；管理员在 VibeAdmin 封禁一个用户，所有端立即生效。

### 2. 统一的账本

积分、充值、消费记录是**全局唯一**的。

- **积分账户** — `point_accounts` 表，每用户一行，四端读写同一余额
- **充值订单** — `recharge_orders` 表，无论从哪个端充值，订单统一入账
- **消费记录** — `api_logs` + `point_transactions` 双表记账，任何端的调用都留下可审计的流水
- **幂等到账** — 支付回调支持幂等，重复回调不会重复充值

### 3. 统一的经营

VibeAdmin 运营后台基于**同一套数据库**进行运营管理。

| VibeBase（用户侧） | VibeAdmin（运营侧） |
| --- | --- |
| 用户注册 | 用户列表 / 封禁 / 角色分配 |
| 用户充值 | 收入统计 / 订单管理 |
| 用户提工单 | 工单处理 / 状态流转 |
| 用户反馈 | 反馈查看 / 回复 |
| AI 能力调用 | 能力定价 / 上下架 |
| 系统公告（接收） | 系统公告（发布） |

### 4. 统一的收钱（支付中台 VibePay）

VibePay 是商业闭环的「收钱」环节，让充值不再依赖官方商户资质：

- **免签约收款** — 无需营业执照，用个人微信/支付宝收款码即可收款，资金直达你的账户
- **自动回调入账** — 安卓监控端监听收款通知栏 → VibePay 匹配订单 → 异步回调 VibeBase，积分自动到账
- **多租户隔离** — 按 `tenant_id` 行级隔离，每商户独立密钥与收款码，支持「公司级 + 应用级」接入模型
- **已部署线上** — [https://pay.vibeadmin.cn/](https://pay.vibeadmin.cn/)，开箱即用

> 没有 VibePay，充值只是「生成订单」；有了 VibePay，充值才是「用户付款、系统收钱、积分到账」的完整生意。

## 端口分配

开发环境下，四端有固定的端口约定（避免冲突）：

| 端 | 前端端口 | 后端端口 | 说明 |
| --- | --- | --- | --- |
| VibeBase | 5175 | 8081 | 用户 Web |
| VibeAdmin | 5173 | 8080 | 运营后台 |
| Vibe-Mp-H5 | 5174 | — | 复用 VibeBase API |
| VibeApp | 5176（Web 预留） | — | Flutter，复用 VibeBase API |
| VibePay | — | 8080（线上 `pay.vibeadmin.cn`） | 支付中台（独立 PostgreSQL 库） |

::: tip 端口约定来源
这套端口分配是单一事实来源，已写入后端 CORS 白名单（`main.py`）与各端文档。修改端口需同步更新 CORS 配置。
:::

## 数据库共享关系

VibeBase 和 VibeAdmin 共享同一 PostgreSQL 实例。核心表的所有权划分：

```
PostgreSQL (vibe 数据库)
│
├── 用户体系（共享）
│   ├── users              ← C 端用户
│   ├── roles              ← 角色定义
│   └── user_roles         ← 用户-角色映射
│
├── 对话体系（VibeBase 专属）
│   ├── dialogs            ← 对话会话
│   ├── histories          ← 消息历史
│   ├── message_likes      ← 点赞
│   └── message_downs      ← 反对
│
├── 计费体系（共享）
│   ├── point_accounts     ← 积分账户
│   ├── point_transactions ← 积分流水
│   ├── recharge_orders    ← 充值订单
│   ├── abilities          ← AI 能力定价
│   └── api_logs           ← 调用日志
│
├── 运营体系（共享）
│   ├── tickets            ← 工单
│   ├── feedbacks          ← 反馈
│   ├── announcements      ← 公告
│   └── operation_logs     ← 操作日志
│
├── 开发体系（VibeBase 专属）
│   ├── api_keys           ← API Key
│   └── sub_accounts       ← 子账号
│
└── 后台体系（VibeAdmin 专属）
    ├── admin_users        ← 后台管理员账号
    ├── tasks              ← 任务
    └── system_config      ← 系统配置
```

## 该用哪个端？

::: tip 决策建议
- **你是开发者，想看产品全貌** → 先跑 VibeBase，再跑 VibeAdmin
- **你只想要一个能对话的 Web** → 只跑 VibeBase（用户 Web + 后端）
- **你要做微信生态** → Vibe-Mp-H5
- **你要做原生 App** → VibeApp
- **你要运营收费** → VibeBase + VibeAdmin 必备
- **你要真正收到钱（免签约）** → 加接 VibePay 支付中台（[pay.vibeadmin.cn](https://pay.vibeadmin.cn/)）
:::
